# Step 04: Background Runners and Auth Flow

Update background runners to handle multi-workspace task processing, and implement the new auth flow with workspace selection. This step depends on Step 03 (application layer) being completed.

---

## 1. Background Runners

Runners poll for pending tasks across ALL workspaces with a single query (no workspace filter). They resolve workspace context from the task's parent chain when processing.

### Current Architecture

Runners are in `backend/runner/` and include:
- `approval/` -- approval workflow processing
- `relay/` -- external approval relay
- `taskrun/` -- task execution scheduler
- `schemasync/` -- database schema sync
- `slowquerysync/` -- slow query log sync
- `mail/` -- email notifications
- `metricreport/` -- metric reporting
- `plancheck/` -- plan check execution

### How Runners Work in SaaS Mode

```go
func (r *Scheduler) Run(ctx context.Context) {
    ticker := time.NewTicker(5 * time.Second)
    for {
        select {
        case <-ticker.C:
            // No workspace filter -- returns pending task runs across all workspaces
            tasks, _ := r.store.ListPendingTaskRuns(ctx)
            for _, task := range tasks {
                go r.processTask(ctx, task)
            }
        }
    }
}

func (r *Scheduler) processTask(ctx context.Context, taskRun *TaskRun) {
    // Resolve workspace through parent chain:
    // task_run -> task -> plan -> project -> workspace_id
    task, _ := r.store.GetTask(ctx, taskRun.TaskID)
    plan, _ := r.store.GetPlan(ctx, &FindPlanMessage{UID: &task.PlanID})
    project, _ := r.store.GetProjectByResourceID(ctx, plan.ProjectID)
    workspaceID := project.WorkspaceID

    // Use workspaceID for workspace-specific operations
    // (e.g., loading instance credentials, checking workspace settings)
}
```

### Runner-Specific Considerations

| Runner | Workspace resolution | Notes |
|--------|---------------------|-------|
| `taskrun/` | task_run -> task -> plan -> project -> workspace_id | Main execution pipeline |
| `plancheck/` | plan_check_run -> plan -> project -> workspace_id | Plan validation |
| `approval/` | issue -> project -> workspace_id | Approval workflows |
| `relay/` | issue -> project -> workspace_id | External approval relay |
| `schemasync/` | instance -> workspace_id (direct) | Instance is a root table |
| `slowquerysync/` | instance -> workspace_id (direct) | Instance is a root table |
| `mail/` | Varies by notification source | Resolve from the entity being notified about |
| `metricreport/` | Reports across all workspaces | May aggregate per-workspace or globally |

### Optimization: Denormalize `workspace_id`

For frequently polled runner queries, consider denormalizing `workspace_id` onto hot tables to avoid the join chain on every poll cycle:

```sql
-- Optional: add workspace_id to task for faster runner lookups
ALTER TABLE task ADD COLUMN workspace_id text REFERENCES workspace(id);
-- Backfill from parent chain: task -> plan -> project -> workspace_id
UPDATE task SET workspace_id = (
    SELECT p.workspace_id FROM project p
    JOIN plan pl ON pl.project = p.resource_id
    WHERE pl.id = task.plan_id
);
```

This is an optimization, not a requirement. Start without it and add if runner query latency becomes an issue.

## 2. Auth Flow Changes

### Current Flow (Per-Workspace Container)

1. User hits `{id}.bytebase.com` -> single workspace container
2. Login with email/password or SSO
3. JWT scoped to workspace implicitly (one workspace per container)

### SaaS Flow (Single Container)

1. User hits `console.bytebase.com`
2. Login (password/SSO) -> authenticates against global `principal` table (END_USER only)
3. Query workspace membership: find all workspaces where the user appears in `policy` (IAM binding with `resource_type = 'WORKSPACE'`)
4. If user has 1 workspace -> auto-redirect
5. If user has N workspaces -> workspace picker UI
6. If user has 0 workspaces -> create workspace or accept invitation
7. After workspace selected -> JWT includes both `user_id` (principal.id) and `workspace_id`
8. All subsequent requests scoped to that workspace

### JWT Structure

```json
{
  "sub": "users/user@example.com",
  "user_id": 123,
  "workspace_id": "ws-abc123",
  "iat": 1709654400,
  "exp": 1709740800
}
```

### Workspace Discovery Query

Find workspaces a user belongs to by scanning IAM policies:

```sql
-- Find all workspaces where user@example.com is a member
SELECT DISTINCT w.id, w.resource_id, w.name
FROM workspace w
JOIN policy p ON p.workspace_id = w.id
WHERE p.resource_type = 'WORKSPACE'
  AND p.type = 'IAM'
  AND EXISTS (
      SELECT 1
      FROM jsonb_array_elements(p.payload->'bindings') AS binding,
           jsonb_array_elements_text(binding->'members') AS member
      WHERE member = 'users/user@example.com'
  )
  AND w.deleted = FALSE;
```

### Service Accounts and Workload Identities

These are workspace-scoped in their own tables (`service_account`, `workload_identity`), each with a `workspace_id` column. They authenticate directly within workspace context (API key or token exchange) -- no workspace picker needed.

### Auth Token Tables

`web_refresh_token`, `oauth2_client`, `oauth2_authorization_code`, `oauth2_refresh_token` remain tied to the `principal` (identity) layer since login is cross-workspace. No `workspace_id` needed on these tables.

## 3. Workspace Switching

After initial login and workspace selection, users may need to switch workspaces without re-authenticating.

### Approach

1. User clicks "Switch Workspace" in the UI
2. Frontend calls a workspace-switch endpoint with the target `workspace_id`
3. Backend validates user is a member of the target workspace (check IAM policy)
4. Backend issues a new JWT with the updated `workspace_id`
5. Frontend reloads with new workspace context

```go
func (s *AuthService) SwitchWorkspace(ctx context.Context, req *SwitchWorkspaceRequest) (*SwitchWorkspaceResponse, error) {
    userID := UserIDFromContext(ctx)

    // Validate user is a member of the target workspace
    iamPolicy, err := s.store.GetWorkspaceIamPolicy(ctx, req.WorkspaceID)
    if err != nil { return nil, err }

    if !isMember(iamPolicy, userID) {
        return nil, status.Errorf(codes.PermissionDenied, "not a member of workspace %s", req.WorkspaceID)
    }

    // Issue new JWT with updated workspace_id
    token, err := s.issueJWT(userID, req.WorkspaceID)
    // ...
}
```
