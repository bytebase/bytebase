# In-App Purchase for SaaS Workspaces

Move the purchase workflow from the hub codebase into the Bytebase codebase so SaaS users can purchase licenses directly from their workspace.

---

## Scope

**In scope**: Stripe checkout, webhooks, subscription CRUD, pricing, license activation — SaaS only.

**Out of scope**: GCP deployment, Paddle, self-host license download, Auth0, Mailchimp, Slack notifications, cloud workspace provisioning.

**Guiding principle**: Non-SaaS hides the purchase UI and rejects purchase APIs. The existing `UpdateSubscription` (manual JWT upload) remains for self-hosted.

---

## Current State

### Bytebase (this codebase)

- **License storage**: JWT string in `setting` table (workspace-scoped), key `(workspace, "bb.workspace.license")`.
- **License parsing**: `enterprise/license.go` — RSA256 JWT with claims `{seat, instanceCount, instance, plan, orgName, workspaceId, ha, trialing}`.
- **LicenseService**: `LoadSubscription(ctx, workspaceID)` → cached `*v1pb.Subscription`. `StoreLicense(ctx, workspaceID, jwt)` → validates + persists.
- **API**: `GetSubscription` (unauthenticated), `UpdateSubscription` (manual JWT upload, `bb.settings.set`).
- **Frontend**: `SettingWorkspaceSubscription.vue` — shows plan info, textarea for JWT upload (self-host only).
- **Webhook infra**: Echo routes under `/hook/` (SCIM already registered here).

### Hub (source of truth for purchase logic)

- **Subscription table**: `id, org, instance_count, seat, interval, status, expires_at, payload(JSONB)`.
- **Stripe checkout**: Creates `CheckoutSession` with dynamic `PriceData`, metadata `{sub_id, plan, interval, instance_count, user_count, source}`.
- **Stripe webhooks**: `customer.subscription.created/updated/paused/resumed/deleted`, `invoice.paid`.
- **Cancellation**: Monthly = immediate proration + refund across charges. Annual = cancel at period end.
- **Subscription update**: Cancel old Stripe sub → create new one with existing customer's payment method.
- **Pricing**: `PlanLimitConfig` per plan type, `PriceModel` calculates total from base + additionals.
- **License generation**: RS256 JWT with same claims structure as Bytebase parser expects.

## Design

### 1. Database: `subscription` Table

Add a new `subscription` table (workspace-scoped). This replaces hub's subscription table with `org` → `workspace`.

```sql
CREATE TABLE subscription (
    id          text        NOT NULL,
    workspace   text        NOT NULL REFERENCES workspace(resource_id),
    status      text        NOT NULL DEFAULT '',    -- 'PENDING', 'ACTIVE', 'PAUSED', 'CANCELED'
    started_at  timestamptz NOT NULL DEFAULT now(),
    expires_at  timestamptz NOT NULL DEFAULT now(),
    payload     jsonb       NOT NULL DEFAULT '{}',  -- SubscriptionPayload proto
    etag        text        NOT NULL DEFAULT '',
    created_at  timestamptz NOT NULL DEFAULT now(),
    updated_at  timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (id),
    UNIQUE (workspace)
);
```

The table is intentionally minimal — only `status`, `started_at`, `expires_at` are top-level columns because they are queried/filtered on. All billing details live in `payload` JSONB so the schema can evolve without DDL migrations.

**Key differences from hub**:
- `workspace` FK instead of `org` FK.
- `seat`, `interval`, `instance_count` moved into `payload` (not top-level columns).
- One subscription per workspace (unique constraint).

**Payload** (proto, stored as JSONB via `protojson.Marshal` → camelCase keys):

```protobuf
// In proto/store/store/subscription.proto

import "v1/subscription_service.proto";

message SubscriptionPayload {
  // Stripe integration
  string stripe_subscription_id = 1;
  string stripe_customer_id = 2;

  // Billing details — enums reuse the v1 API types
  bytebase.v1.PlanType plan = 3;
  BillingInterval interval = 4;
  int32 seat = 5;
  int32 instance_count = 6;     // fixed at 10 for TEAM, but stored for license generation

  enum BillingInterval {
    BILLING_INTERVAL_UNSPECIFIED = 0;
    MONTH = 1;
    YEAR = 2;
  }
}
```

Note: `plan` reuses the existing `bytebase.v1.PlanType` enum (`FREE`, `TEAM`, `ENTERPRISE`) from `subscription_service.proto`. `interval` uses a local `BillingInterval` enum since no equivalent exists in v1 today. `protojson.Marshal` serializes enums as their string names (e.g., `"TEAM"`, `"MONTH"`) in JSONB.

No `DeploymentType` (always SaaS). No `PaddlePayload`. No `Source` enum (always Stripe). Billing fields in the payload keep the table lean and make it easy to add new fields (e.g., future per-instance pricing) without schema changes.

**Why separate `CreatePurchase`/`UpdatePurchase`/`CancelPurchase` from existing `UpdateSubscription`?**
- `UpdateSubscription` is the self-hosted JWT upload path (will be disabled in SaaS).
- Purchase RPCs are SaaS-only and involve Stripe interaction.
- Clean separation — no conditional logic mixing two unrelated flows.


**Event handlers** (ported from hub `webhook/stripe.go`):

| Event | Handler |
|-------|---------|
| `customer.subscription.created` | Extract metadata, update subscription with Stripe IDs |
| `customer.subscription.updated` | Map Stripe status → internal status |
| `customer.subscription.paused` | Set PAUSED, invalidate license cache |
| `customer.subscription.resumed` | Set ACTIVE, regenerate + store license |
| `customer.subscription.deleted` | Set CANCELED, clear license |
| `invoice.paid` | Update seats/interval from metadata, extend `expires_at`, generate + store license JWT |

**License activation on `invoice.paid`** (replaces hub's GCP injection):

```go
func (h *WebhookHandler) activateLicense(ctx context.Context, sub *SubscriptionMessage) error {
    // 1. Build license claims from subscription data
    claims := &enterprise.Claims{
        Seats:           sub.Seats,
        ActiveInstances: 10, // fixed for TEAM
        Instances:       10,
        Plan:            sub.Payload.Plan,
        WorkspaceID:     sub.Workspace,
    }
    // 2. Sign JWT using the license private key
    jwt := h.licenseService.CreateLicense(claims)
    // 3. Store directly via LicenseService
    h.licenseService.StoreLicense(ctx, sub.Workspace, jwt)
    // → This updates the setting table AND invalidates the cache
}
```

This is the key simplification: instead of generating a JWT on a remote hub and injecting it via GCP API, we generate and store it in-process.

### 9. License Key Management

**Problem**: Hub uses a private key to sign license JWTs. Bytebase uses a public key to verify them. For in-app purchase, Bytebase needs the private key to sign licenses.

**Solution**: Add a `prod.priv.pem` (and `dev.priv.pem`) to `backend/enterprise/keys/`, loaded only when `profile.SaaS == true`.

```go
type Config struct {
    publicKey  *rsa.PublicKey
    privateKey *rsa.PrivateKey  // nil for self-hosted
}

// CreateLicense signs a new license JWT. Only available in SaaS mode.
func (s *LicenseService) CreateLicense(claims *claims) (string, error) {
    if s.config.privateKey == nil {
        return "", errors.New("license signing not available")
    }
    token := jwt.NewWithClaims(jwt.SigningMethodRS256, claims)
    token.Header["kid"] = "v1"
    return token.SignedString(s.config.privateKey)
}
```

The private key should be provided via environment variable (not embedded in binary) for security:
- `LICENSE_PRIVATE_KEY` — PEM-encoded RSA private key, loaded only when `profile.SaaS == true`.

## Key Differences from Hub

| Aspect | Hub | Bytebase SaaS |
|--------|-----|----------------|
| License delivery | JWT download (self-host) or GCP injection (cloud) | In-process `StoreLicense()` |
| Instance pricing | Per-instance additional | Fixed 10 instances (TEAM) |
| Payment providers | Stripe + Paddle | Stripe only |
| Grace period | +7 days (self-host), +12h (cloud) | None needed (instant activation) |
| Subscription owner | `org` entity | `workspace` directly |

## Security Considerations

1. **Stripe webhook signature verification** — Must validate `Stripe-Signature` header against `STRIPE_WEBHOOK_SECRET`. Reject unsigned requests.
2. **License private key** — Environment variable only, never embedded in binary. Only loaded in SaaS mode.
3. **SaaS gate** — All purchase APIs must check `profile.SaaS` before proceeding. Non-SaaS deployments must never expose Stripe integration.
4. **Etag-based optimistic locking** — Prevents race conditions on subscription updates.
5. **Metadata validation** — Webhook handler must verify `source=bytebase_hub` in metadata to filter out unrelated Stripe events.
6. **HTTPS only** — Stripe webhooks require HTTPS endpoints.
