## High Priority — Not Yet Implemented

### Workspace-scoped resource listing

All List/Search/Get endpoints for project-child tables must validate the parent project belongs to the user's workspace. Currently, child table queries filter by project FK but don't verify the project's workspace, which could leak data across workspaces.

**Affected endpoints and store queries:**

| Category | Endpoints | Risk |
|----------|-----------|------|
| **Databases** | `ListDatabases`, `BatchGetDatabases`, `GetDatabase` | `FindDatabaseMessage` doesn't include workspace. Queries by project/instance which are workspace-scoped in store, but API doesn't validate parent workspace. |
| **Issues** | `ListIssues`, `SearchIssues`, `GetIssue` | `FindIssueMessage` has `ProjectIDs` but no workspace. Project-level filter is set from request `parent` without workspace validation. |
| **Plans** | `ListPlans`, `GetPlan` | `FindPlanMessage` queries by `ProjectID` without workspace validation on the project. |
| **Database Groups** | `ListDatabaseGroups`, `GetDatabaseGroup` | `FindDatabaseGroupMessage` has no workspace. `ListDatabaseGroups` in actuator queries with `{}` (all workspaces). |
| **Changelogs** | `ListChangelogs` | Queried by database which is instance-scoped, but no workspace validation on instance. |
| **Releases** | `ListReleases` | Queried by project without workspace validation. |
| **Worksheets** | `ListWorksheets` | Queried by project/creator without workspace validation. |
| **Task Runs** | `ListTaskRuns` | Queried by plan/project without workspace validation. |
| **Access Grants** | `ListAccessGrants` | Queried by project without workspace validation. |
| **Query History** | `SearchQueryHistories` | Queried by creator without workspace validation on project. |

**Fix pattern:** For each child List/Get endpoint:
1. Extract `projectID` (or `instanceID`) from request `parent`
2. Validate the parent belongs to the current workspace: `GetProject(ctx, FindProjectMessage{Workspace: wsID, ResourceID: &projectID})`
3. Only then query children

**Also need:** Add workspace to `FindDatabaseGroupMessage` (it's a project-child but currently has no workspace-aware parent validation in the store).

## Design Decisions

1. **`server_config` only stores `auth_secret`** — All other settings (password restrictions, signup control, token durations, metrics, debug) live in per-workspace WORKSPACE_PROFILE. Reason: avoid split-brain between two config sources. The login page can get restrictions from the workspace-scoped actuator endpoint.

2. **Root table PKs (`resource_id`) are globally unique** — No composite PK migration needed. Workspace-scoped unique indexes only for tables where PK is not `resource_id` (setting, user_group, service_account, workload_identity, policy).

3. **Three access patterns for store queries**:
   - **API layer**: `common.GetWorkspaceIDFromContext(ctx)` from JWT
   - **Runners**: Cross-workspace methods (`GetProjectByResourceID`, `ListAllInstances`)
   - **Login/startup**: `store.GetWorkspaceID(ctx)` or `FindWorkspacesByMemberEmail`

4. **OAuth2 child tables** (auth code, refresh token) don't need workspace — scoped through `client_id` FK to workspace-scoped `oauth2_client`.
